# AI Factory Roadmap

> Informed by a review of emerging AI-native development workflows — cognitive mode separation, automated shipping, browser-based QA, and data-driven retrospectives.

---

## Bucket 1: Short-term (enhance existing factory workflows)

### 1.1 — "Product Taste" pre-spec challenge skill

Before `/opsx:propose`, run a skill that pressure-tests the idea with a "10-star product" lens. Asks: is this the real feature, or the obvious one? What would make this feel inevitable? Challenges the Product Owner to think bigger before committing to a spec.

### 1.2 — Enhanced ship workflow

Upgrade the shipping workflow with: auto-generated changelog from commit history, smarter commit splitting (infrastructure > models > views > version), auto version-bump heuristics (patch/minor based on diff size). Reduce friction in the last mile.

### 1.3 — Pre-landing structural review

Add a "paranoid staff engineer" review pass that runs before shipping — not a code review against a plan, but a structural audit looking for: race conditions, missing error handling, trust boundary violations, tests that pass while missing the real failure mode. A different posture to the existing code-review skill.

### 1.4 — Factory retrospective skill

A retrospective skill that analyses git history across factory projects. Track: velocity trends per project, test coverage over time, subagent success rates, context-clear frequency, session patterns. Save JSON snapshots for trend comparison. Adapted for solo factory context.

### 1.5 — Cognitive mode concept in CLAUDE.md

Formalise the idea that within Execution Mode, the AI can adopt different cognitive postures (builder, reviewer, debugger, shipper) rather than one undifferentiated "engineer brain." The Superpowers skills already do some of this (TDD, debugging, code review) but making it explicit could improve discipline.

## Bucket 2: Medium-term (new capabilities for web products)

### 2.1 — Browser QA stack profile

Create `stacks/browser-qa/` that documents how to integrate headless browser QA tools into factory web projects. Covers: when to use browser QA vs unit tests, integration with the factory iteration loop, cookie/auth setup for authenticated testing.

### 2.2 — Web product template with QA integration

Extend `templates/ai-product-template/` (or create a web-specific variant) that includes browser QA in the iteration loop and the "give the agent eyes" pattern for visual verification.

### 2.3 — Template system for stack profiles

Auto-generate parts of stack profile docs from project configs to prevent drift. Static validation tests to catch stale docs. Reduces maintenance burden as more stacks are added.

### 2.4 — Diff-aware QA in factory workflow

For web projects, after Superpowers implements a task, automatically run diff-aware QA (analyse changed routes, test affected pages) before marking the task complete. Bridges the gap between "tests pass" and "it actually works."

## Bucket 3: Long-term / Vision

### 3.1 — Multi-session orchestration

Run multiple Claude Code sessions in parallel, each with a different cognitive mode. Factory equivalent: spec session, design session, and multiple implementation sessions working on different tasks simultaneously. Currently we do subagent parallelism within one session — this would be session-level parallelism.

### 3.2 — Factory analytics dashboard

Persistent tracking of factory output across all projects. Products shipped, total tests, lines of code, time-to-MVP, quality metrics. The "how productive is the factory" question answered with data, not vibes.

### 3.3 — Visual regression testing integration

For web products: screenshot baselines, pixel-diff comparisons across deploys, automated layout regression detection. Adapted for the factory's continuous iteration loop.

### 3.4 — Automated deployment verification

Post-deploy smoke tests that verify the shipped product actually works in production. A post-ship verification step in the iteration loop.

### 3.5 — Design Mode visual verification

Connect Design Mode deliverables to browser QA — after implementing a design spec, automatically compare the running app against the design tokens/mockups. "Does the built thing match the designed thing?"

### 3.6 — Swarm Protocol as factory coordination layer

Multi-session orchestration (3.1) provides the runtime — run N agents in parallel. A coordination protocol provides the missing layer: who's working on what, what's blocked, what just finished. Most immediate value in daughter projects where multiple agents work on the same codebase, replacing manual subagent task assignment with decentralised self-coordination. Also enables multi-person factory use if AI Factory grows beyond solo operation.
